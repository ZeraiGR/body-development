# Архитектура и техническая документация

Подробное описание проекта «Бот реабилитации спины». Для быстрого старта см. [README.md](README.md);
поведение и тон бота — в [CLAUDE.md](CLAUDE.md); стратегия и тактика — в [YEARLY_PLAN.md](YEARLY_PLAN.md)
и [MONTH_1_PLAN.md](MONTH_1_PLAN.md). Здесь — как именно это реализовано.

---

## 1. Назначение и границы

**Что это.** Личный Telegram-бот-ассистент по реабилитации спины для одного пользователя
(разработчик, сидячий образ жизни). Ведёт 30-дневную программу: утром даёт теорию и утренний
запуск, днём напоминает о микро-паузах, вечером собирает метрики и поддерживает мотивацию
научно-обоснованным фидбеком.

**Что это НЕ.** Не медицинское устройство, не диагностика, не публичный сервис. Не содержит LLM —
все ответы шаблонные и детерминированные. Не работает в группах.

---

## 2. Стек и обоснование

| Компонент | Технология | Почему |
|-----------|------------|--------|
| Бот | aiogram 3.x | Требование ТЗ (CLAUDE.md §8); современный async-фреймворк Telegram |
| Расписание | APScheduler (AsyncIOScheduler) | Cron-задачи утром/днём/вечером (ТЗ §8) |
| Хранение | SQLite (aiosqlite) | ТЗ §8; один пользователь → локальный файл достаточен |
| Теория | Telegraph API (aiohttp) | ТЗ §8; статьи-hosting без своего сервера |
| Конфиг | python-dotenv | Секреты вне кода |
| Ответы | Шаблоны на чистом Python | Решение пользователя: детерминированно, дёшево, без API-ключа LLM |

Режим запуска — **long-polling, локально** (решение пользователя): не нужен публичный URL,
webhook и Docker.

## 3. Слои и модули

```
main.py                      точка входа: сборка бота, диспетчера, middleware, планировщика
│
├── config.py                env → неизменяемый Config (токены, chat_id, таймзона, тайминги, БД)
│
└── bot/
    ├── security.py          AuthMiddleware — единственный рубеж допуска (single-user)
    ├── db.py                SQLite: схема + async-хелперы
    ├── planner.py           логика программы: перевод дня, недели, стадии, streak, sparkline
    ├── messages.py          тексты и клавиатуры (формула фидбека §5)
    ├── handlers.py          команды + колбэки вечернего флоу (aiogram Router)
    ├── scheduler.py         APScheduler: утро / пинг / вечер / воскресный отчёт
    ├── telegraph.py         клиент Telegraph API (aiohttp)
    └── content/
        ├── articles.py      30 статей теории (данные)
        └── program.py       недели (MONTH_1_PLAN) + научные этапы (CLAUDE.md §3) (данные)
```

**Граф зависимостей (кто кого импортирует):**

```
main ─┬─► config
      ├─► db
      ├─► handlers ─► db, messages, planner, config
      ├─► scheduler ─► db, planner, messages, config
      └─► security ─► (aiogram)

planner ─► content.program
messages ─► content.program, content.articles, planner
content.program, content.articles ─► (ничего, чистые данные)
scripts/upload_telegraph ─► config, db, content.articles, telegraph
```

Чистые данные (`articles`, `program`) не зависят ни от чего — их можно тестировать и менять
изолированно. `planner` и `messages` — чистые функции над данными (легко покрываются тестами).
Побочные эффекты (Telegram, БД, сеть) сосредоточены в `handlers`, `scheduler`, `db`, `telegraph`.

## 4. Модель данных

SQLite, три таблицы. Однопользовательский режим → состояние в единственной строке.

### `user_state` (ровно одна строка, `id = 1`)

| Поле | Тип | Назначение / инвариант |
|------|-----|------------------------|
| `id` | int PK | Всегда 1 (`CHECK (id = 1)`) |
| `current_day` | int | День программы 1..30. Переводится только утренним rollover |
| `streak` | int | Подряд идущих дней с вечерним логом |
| `paused` | int (0/1) | Пауза: замораживает рассылки и перевод дня |
| `week_extra_days` | int | Сколько дней «задержаться на неделе»: rollover сжигает по одному вместо перевода |
| `last_morning_date` | text | Дата последнего утреннего запуска — ключ идемпотентности rollover |
| `last_log_date` | text | Дата последней вечерней отметки — для streak |
| `started_at` | text | Дата первого старта |

### `daily_logs` (строка = календарный день)

| Поле | Тип | Назначение |
|------|-----|-----------|
| `date` | text PK | `YYYY-MM-DD` |
| `day_number` | int | Какой это был день программы |
| `pain` | int? | Боль 1..10 (NULL, если не отвечал) |
| `stiffness` | int? | Жёсткость правой стороны 1..10 |
| `morning_done` | int (0/1) | Нажал «✅ Сделал» утром |
| `evening_done` | int (0/1) | Завершил вечерний флоу |
| `habits_done` | int (0/1) | Выполнил вечерний ритуал |
| `note` | text | Зарезервировано (сейчас не пишется хендлерами) |

Индекс `idx_logs_date` по `date` — для оконных запросов отчётов.

### `telegraph_links`

| Поле | Тип | Назначение |
|------|-----|-----------|
| `day` | int PK | 1..30 |
| `url` | text | URL статьи на telegra.ph |
| `title` | text | Заголовок статьи |

**Доступ к данным** — строго через параметризованные запросы (`?`-плейсхолдеры в `db.py`).
Пользовательский текст никогда не подставляется в SQL.

## 5. Жизненный цикл дня

### Утро (`MORNING_TIME`, по умолчанию 08:30, таймзона `TIMEZONE`)

1. `if state.paused → return` (полный фриз).
2. **Rollover** (`planner.morning_rollover`) — см. §6. Может перевести `current_day`.
3. Отправка `messages.morning_text(day, telegraph_url)`: вода+дыхание, заголовок и ссылка
   статьи дня, 🟢 обязательно / 🟡 опционально, утренние действия недели, кнопки
   `✅ Сделал / ⏰ Позже`.
4. Если сегодня воскресенье — дополнительно недельный отчёт (`weekly_report_text`).

Кнопка `morning:done` → `daily_logs.morning_done = 1`. На streak не влияет (он вечерний).

### День (1 случайный момент в окне `DAY_PING_START..DAY_PING_END`, по умолчанию 13:00–17:00)

1. `asyncio.sleep(randint(0, длина_окна_в_сек))`.
2. `if state.paused → return`.
3. Случайный вариант микро-паузы (`day_ping_text`) + кнопки `✅ Сделал / ⏩ Пропустить`.

### Вечер (`EVENING_TIME`, по умолчанию 20:30)

1. `if state.paused → return`.
2. Отправка `evening_intro_text`: интеграция с Дотой/сериалом + аппликатор, вечерний ритуал
   недели, вопрос «Как спина сегодня по 10-балльной шкале?» + клавиатура `pain:1..10`.

**Флоу опроса (на колбэках, состояние — в БД за сегодняшний день):**

```
pain:N   → upsert_log(pain=N)        → edit → вопрос о жёсткости (stiff:1..10)
stiff:N  → upsert_log(stiffness=N)   → edit → вопрос о ритуале   (habits:yes/partial/no)
habits:X → upsert_log(habits_done, evening_done=1)
           → update_streak → update_state(streak, last_log_date)
           → edit → feedback_text (формула §5) + кнопка «🥵 Было тяжело»
hard     → week_extra_days = 2       → подтверждение задержки на неделе
```

## 6. Конечный автомат программы

### Перевод дня (`morning_rollover`) — идемпотентный

```
если paused                         → {} (ничего; причём _morning вообще не вызовет отправку)
если last_morning_date == сегодня    → {} (уже запускалось сегодня — рестарт-устойчиво)
если last_morning_date is None       → {last_morning_date: сегодня}        ← САМЫЙ ПЕРВЫЙ старт, день не двигается
иначе:
    если week_extra_days > 0         → {week_extra_days: -1}               ← задержка на неделе
    иначе если current_day < 30      → {current_day: +1}                   ← обычный перевод
    иначе (current_day == 30)        → {}                                  ← программа завершена
    + {last_morning_date: сегодня}
```

**Свойства:** первый запуск показывает день 1 (без прыжка на 2); рестарт бота днём не дёргает
счётчик; пауза замораживает; «тяжело» сдвигает перевод, не ломая идемпотентность.

### Streak (`update_streak`) — считается при завершении вечернего флоу

```
если last_log_date == сегодня → (streak без изменений, changed=False)   ← повторная отправка
если last_log_date == вчера   → streak += 1
иначе (пропуск/первый раз)    → streak = 1
```

Пропущенный вечер не «наказывается» сообщением — streak естественно обнулится при следующем логе.

### «Задержаться на неделе»

Кнопка `hard` или команда `/hard` ставит `week_extra_days = 2`. Два следующих утра rollover
сжигает по одному дню, не переводя `current_day` → пользователь остаётся в текущей неделе ещё
2 дня, как требует MONTH_1_PLAN/CLAUDE.md.

### Пауза

`/pause` → `paused = 1`. Все три cron-задачи возвращаются досрочно; rollover не переводит день.
`/resume` → `paused = 0`.

## 7. Планировщик

`BotScheduler(bot, chat_id)` создаёт `AsyncIOScheduler(timezone=config.schedule.timezone)` и три
задачи с `CronTrigger`:

| id | Триггер | Действие |
|----|---------|----------|
| `morning` | `MORNING_TIME` | rollover + утреннее сообщение (+ воскресный отчёт) |
| `day_ping` | `DAY_PING_START` | случайная задержка + микро-пауза |
| `evening` | `EVENING_TIME` | старт вечернего опроса |

Таймзона берётся из `config.schedule.timezone` (`Europe/Moscow` по умолчанию) — все времена
интерпретируются одинаково. `replace_existing=True` делает перезапуск безопасным. Задача
`day_ping` считается внутри окна, поэтому рестарт после `DAY_PING_START` может пропустить пинг
этого дня — приемлемо для личного бота.

## 8. Модель безопасности

Требование пользователя: **бот общается только с владельцем и защищён от промпт-инъекций.**

### 8.1. Single-user guard

`AuthMiddleware` (наследник `aiogram.BaseMiddleware`) зарегистрирован как **outer-middleware**
одновременно на `dp.message` и `dp.callback_query` — единственная точка пропуска. Логика:

- `event.from_user.id != ALLOWED_CHAT_ID` → отбросить (`return`, без вызова хендлера);
- `chat.type != "private"` → отбросить (группы/каналы запрещены даже владельцу);
- иначе → `await handler(event, data)`.

Чат определяется с поддержкой обоих типов событий: у `Message` это `event.chat`, у
`CallbackQuery` — `event.message.chat`. Попытки чужого доступа логируются (`WARNING`).

Прочие типы апдейтов (`edited_message`, `channel_post`, `my_chat_member`, …) просто не имеют
зарегистрированных хендлеров и игнорируются aiogram.

### 8.2. Устойчивость к промпт-инъекциям

Поверхности атаки **нет по архитектуре**: LLM отсутствует, поэтому пользовательский текст нигде
не интерпретируется как инструкция. Конкретные гарантии:

- **Свободный текст игнорируется.** Бот реагирует только на конкретные команды и кнопки;
  неизвестный текст → мягкая подсказка `/help` (без интерпретации).
- **Строгий разбор колбэков.** `parse_rating` принимает только `prefix:int` с `int ∈ [1..10]`;
  `habits` — whitelist `yes/partial/no`. Кривые/неизвестные `callback_data` игнорируются.
- **Никакого исполнения пользовательских данных:** в коде нет `eval`/`exec`/`os.system`/`pickle`,
  пользовательский текст не попадает в URL, команды оболочки или SQL.
- **SQL параметризован** (`?`-плейсхолдеры) — SQL-инъекции невозможны.

### 8.3. Если позже добавится LLM (рекомендация)

Пользовательский текст обязан передаваться как **данные** (user-message), а не как часть
системного промпта; модель не должна получать права вызывать инструменты по содержанию
сообщения; шаблонный «скелет» (расписание, кнопки, сбор метрик) остаётся детерминированным, а
LLM — только для генерации мотивационного фидбека. Это сохранит текущую защиту.

## 9. Модель контента и согласование трёх документов

Источников контента три, и они согласованы по разным осям:

| Источник | Ось | Откуда | Где используется |
|----------|-----|--------|------------------|
| `articles.py` (30 шт.) | **день** | написаны по тематике CLAUDE.md §6 и этапам §3 | утренняя статья, 🟢/🟡 факты |
| `program.WEEKS` (4 шт.) | **неделя** | MONTH_1_PLAN | утренние/вечерние действия, фокус |
| `program.STAGES` (5 диапазонов) | **этап дня** | CLAUDE.md §3 | научный факт в вечернем фидбеке |

**Согласование:**
- Утро: теория = статья дня; действия = `WEEKS[week_for_day(day)]`; показывается название этапа.
- Вечер: научный факт = `STAGES[stage_for_day(day)]`; совет-привязка = по неделе; дельта
  жёсткости = сравнение с вчерашним логом.
- Неделя выводится из дня: 1–7→1, 8–14→2, 15–21→3, 22–30→4 (по MONTH_1_PLAN).

### Разрешение конфликта в ТЗ

В CLAUDE.md §6 упоминается «массив из 30 ссылок» (по статье в день), а правило синхронизации
(§0) и MONTH_1_PLAN указывают на 4 статьи — по неделе. По решению пользователя реализовано **30
статей по дню** для теории, а действия всё равно берутся по неделе. Конфликт снят разделением
осей: теория — день, действия — неделя.

## 10. Тон и формулы

Тон (CLAUDE.md): дружелюбный, системный, без давления, evidence-based. Реализован в текстах
`messages.py`. Вечерний фидбек строится по формуле §5:

```
[Подтверждение усилий]  +  [Научный факт текущего этапа]  +  [Связь с привычкой]
```

- **Усилия:** зависят от streak («…уже N дней подряд»).
- **Оценка боли:** ≤3 — отличная динамика; ≤6 — умеренная; иначе — предложение задержаться.
- **Наука:** факт этапа дня (§3).
- **Дельта жёсткости:** если правая сторона стала мягче — поощрение; если жёстче — «бывает».
- **Привязка к привычке:** совет, зависящий от недели (аппликатор → мостик → мячик → рутина).

Склонение дней по правилам русского языка (`_plural_days`): 1 день / 2–4 дня / 5+ дней, с
корректным исключением 11–14.

## 11. Журнал архитектурных решений

1. **Single-user.** Нет онбординга/регистрации; доступ строго по `ALLOWED_CHAT_ID`. Упрощает
   схему и повышает безопасность.
2. **30 статей по дню** (решение пользователя). Написаны и заливаются на Telegraph скриптом.
3. **Локально, long-polling.** Без сервера, webhook и Docker. Код развёртывание-агностик.
4. **Только правила/шаблоны** (решение пользователя). Без LLM, без внешних API, детерминированно.
5. **Разрешение конфликта 4 vs 30** — разделением осей (§9).
6. **`ParseMode.MARKDOWN` (legacy).** Тексты контролируемые (без случайных `_`/`*`), поэтому риск
   низкий. Известный компромисс — см. §13.

## 12. Что проверено

Передача в эксплуатацию сопровождалась smoke-тестом (удалён после прогона), покрывшим:

- целостность 30 статей и маппинг день→неделя→этап для всех дней 1..30;
- построение утреннего, вечернего сообщения и фидбека для всех 30 дней и вариантов метрик;
- `sparkline`, `stats`, склонение `_plural_days` (включая 11/21);
- логику rollover: первый старт (без прыжка), перевод на день 2, идемпотентность same-day;
- streak: первый лог = 1, повторный лог в тот же день не удваивает;
- круг БД: `upsert_log`/`get_log`/`logs_between`, `set/get_telegraph_link`;
- построение недельного отчёта и `/status`;
- **AuthMiddleware:** владелец+приват пропущен; чужой пользователь отброшен; групповой чат
  отброшен даже для владельца; колбэк-событие (чат через `message`) обработано;
- чистый импорт `handlers`/`scheduler`/`main`.

Результат: **28/28 проверок зелёные.**

## 13. Известные ограничения и компромиссы

- **Markdown legacy.** Нечётное количество `_`/`*` в тексте сломает отправку. Контент
  контролируемый, но при правках статей нужно следить. Альтернатива — переход на HTML.
- **Токен Telegraph не хранится в БД.** Скрипт `upload_telegraph.py` печатает его; нужно вручную
  положить в `.env` (`TELEGRAPH_TOKEN`), иначе повторный запуск создаст новый аккаунт/страницы.
- **Нет дедупликации статей.** Повторный `upload_telegraph.py` создаёт новые страницы и
  перезаписывает ссылки в БД.
- **День программный, не календарный.** День 30 не обязательно воскресенье; «итог месяца»
  привязан к rollover дня 30, а воскресный отчёт — к календарю. Они независимы.
- **Один дневной пинг**, хотя CLAUDE.md допускает 1–2. Расширяемо.
- **Жёсткая локализация RU**, нет i18n.
- **Без FSM.** Состояние вечернего флоу хранится в БД (по дате). Для одного пользователя это
  надёжнее и рестарт-устойчивее; при переходе на multi-user потребуется FSM с ключом по user_id.
- **`note` зарезервирован**, но хендлерами не пишется (свободный текст намеренно не собирается —
  часть модели защиты от инъекций).

   реабилитации, точно следующий трём исходным документам. Логика дней/недель/этапов, streak,
   рассылки, сбор метрик и фидбек — реализованы и покрыты smoke-тестами (28/28).

2. **Конфликт в ТЗ снят чисто.** Разделение осей (теория — день, действия — неделя, наука — этап)
   устранило противоречие «4 vs 30 статей» без потери смысла ни одного документа.

3. **Безопасность — по архитектуре.** Single-user enforcement через единый outer-middleware
   (отброс чужих пользователей и групп) плюс отсутствие LLM (= отсутствие поверхности для
   промпт-инъекций), строгий разбор колбэков и параметризованный SQL. Требование пользователя
   выполнено на уровне дизайна, а не патчей.

4. **Развёртывание-агностик.** Local-first long-polling, но код не привязан к среде — переход на
   сервер/webhook требует изменений только в `main.py` и `scheduler.py`.

5. **Честные компромиссы.** Главные: хрупкость Markdown-парсинга и ручное сохранение токена
   Telegraph. Оба фиксятся точечно (HTML + kv-таблица) и не влияют на архитектуру.

6. **Рекомендация к ежедневному использованию.** Перед стартом стоит: (а) переключить парсинг на
   HTML, (б) один раз залить статьи и сохранить токен. После этого бот production-ready для
   личного применения и готов к наполнению контентом Фазы 2.
